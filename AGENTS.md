# Repository Guidelines For AI Agents

## Purpose
This repository manages personal dotfiles for WSL, Windows, Neovim, WezTerm, shell tooling, and a small Rust CLI named `dotfiles`.

Agents should optimize for safe, incremental edits. Prefer changing one subsystem at a time, verify the specific area you touched, and avoid triggering machine-wide setup unless the user explicitly asks for it.

## Repository Map
- `home/`: user-facing config files that are later symlinked into `$HOME`
- `home/zshrc`, `home/p10k.zsh`, `home/zenhan.zsh`: shell startup and prompt config
- `home/config/nvim/`: Neovim config, mostly Lua plugin specs and core settings
- `home/config/wezterm/`: WezTerm config split across several Lua modules
- `home/config/broot/`, `home/config/lazygit/`, `home/config/lazydocker/`: tool-specific config
- `sh/installers/apt/`, `sh/installers/brew/`, `sh/installers/others/`: bootstrap scripts and package installers
- `ahk/`: AutoHotkey and MouseGestureL config for Windows
- `powershell/`: PowerShell helpers used from WSL during setup
- `rust/`: source for the `dotfiles` CLI
- `command/dotfiles`: built CLI artifact committed to the repo; do not edit this file directly
- `init.sh`: top-level setup entrypoint that runs the CLI

## How This Repo Works
- The Rust CLI performs setup tasks such as symlink creation, installer execution, and shell setup.
- Files under `home/` are the source of truth. They are linked into the user's real home directory by `dotfiles init`.
- `init.sh` and `rust/src/commands/init.rs` have system-level side effects:
  - package installation
  - symlink creation in `$HOME`
  - default shell changes
  - Windows-side WezTerm link creation when running under WSL
- Neovim plugins are configured as separate Lua modules under `home/config/nvim/lua/plugins/`.

## Common Tasks
- Adjust shell aliases, PATH, completions, or interactive shell behavior in `home/zshrc`
- Add or modify Neovim plugin specs under `home/config/nvim/lua/plugins/`
- Update Neovim keymaps, options, or autocmds under `home/config/nvim/lua/core/`
- Tweak WezTerm behavior under `home/config/wezterm/`
- Update package bootstrap scripts under `sh/installers/`
- Modify Rust CLI behavior in `rust/src/` and keep tests passing
- Update Windows automation in `ahk/` or `powershell/`

## Editing Rules For Agents
- Prefer the smallest change that satisfies the request.
- Keep changes scoped to one subsystem unless the user explicitly asks for cross-cutting work.
- Do not edit `command/dotfiles` by hand. If the Rust CLI changes and the user wants the compiled artifact updated, rebuild it from `rust/`.
- Do not casually modify `home/config/nvim/lazy-lock.json`. Only update it when intentionally changing plugin versions or after a plugin management action that requires the lockfile to change.
- Treat `init.sh`, `rust/src/commands/init.rs`, installer scripts, and shell startup files as high-impact files. Small mistakes there affect the whole environment.
- Avoid introducing new external dependencies unless the user asked for them or they are clearly required.
- Preserve existing naming and layout conventions. This repo favors descriptive file names and tool-specific modules.
- When changing shell or Lua config, prefer matching the local style over imposing a new pattern.

## Verification Matrix
Run the checks that match the files you changed.

- Rust:
  - `cd rust && cargo test`
  - `cd rust && cargo build`
- Rust formatting-only changes or simple source changes:
  - `cd rust && cargo fmt --check`
- Neovim Lua:
  - `stylua home/config/nvim`
  - `nvim --headless "+qa"` if startup validation is needed and the environment supports it
- Shell scripts:
  - `bash -n <file>`
- `home/zshrc`:
  - `zsh -n home/zshrc`
- PowerShell scripts:
  - Validate syntax if PowerShell is available; otherwise state that validation was not run
- AutoHotkey files:
  - Static review only unless the user asks for Windows-side runtime validation

If a check cannot run in the current environment, say so explicitly in the final response.

## Task-Specific Guidance

### Shell And Bootstrap
- `home/zshrc` is interactive shell startup code. Regressions here are very visible.
- Installer scripts in `sh/installers/` should remain non-interactive where practical and easy to re-run.
- Avoid running `bash ./init.sh` unless the user explicitly asks, because it installs packages and changes the machine state.

### Neovim
- Plugin specs live under `home/config/nvim/lua/plugins/`.
- Core settings live under `home/config/nvim/lua/core/`.
- Formatting is handled through `conform.nvim`; Lua formatting should follow `.stylua.toml`.
- Keymaps are dense and customized. Check for collisions before adding new mappings.

### Rust CLI
- The CLI entrypoint is `rust/src/main.rs`.
- Commands are implemented under `rust/src/commands/`.
- Shared helpers live under `rust/src/modules/`.
- Add or update tests when changing CLI behavior or filesystem helpers.

### Windows Integration
- `ahk/` and `powershell/` are Windows-oriented and may not be runnable from a Linux-only environment.
- Changes that depend on WSL path translation or Windows symlinks should be reviewed carefully for path format assumptions.

## High-Risk Files And Areas
- `init.sh`
- `rust/src/commands/init.rs`
- `home/zshrc`
- `sh/installers/**/*`
- `powershell/mklink.ps1`
- `home/config/nvim/lazy-lock.json`
- `command/dotfiles`

Prefer extra caution, narrower edits, and explicit verification notes when touching these.

## Commands Agents Should Know
- `bash ./init.sh`
- `dotfiles -h`
- `dotfiles config nvim`
- `cd rust && cargo test`
- `cd rust && cargo build`
- `cd rust && cargo run -- -h`
- `stylua home/config/nvim`

## Commit Style
- Follow the existing short typed subject style:
  - `add: ...`
  - `modify: ...`
  - `fix: ...`
  - `feature: ...`
  - `chore: ...`

Keep commits focused on one logical change.

## Final Response Expectations For Agents
- State what changed in plain language.
- Mention the exact verification you ran.
- Mention any verification you could not run.
- Call out side effects or follow-up steps when editing bootstrap, shell, Windows, or plugin-management code.
