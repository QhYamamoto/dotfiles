# Repository Rules

## Purpose
This repository manages personal dotfiles for WSL, Windows, Neovim, WezTerm,
shell tooling, and a small Rust CLI named `dotfiles`.

Prefer safe, incremental edits. Keep changes scoped to one subsystem unless the
user explicitly asks for cross-cutting work.

## Repository Map
- `home/`: user-facing config files that are later symlinked into `$HOME`
- `home/zshrc`, `home/p10k.zsh`, `home/zenhan.zsh`: shell startup and prompt config
- `home/config/nvim/`: Neovim config, mostly Lua plugin specs and core settings
- `home/config/wezterm/`: WezTerm config split across several Lua modules
- `home/config/broot/`, `home/config/lazygit/`, `home/config/lazydocker/`: tool-specific config
- `sh/installers/apt/`, `sh/installers/brew/`, `sh/installers/others/`: bootstrap scripts and package installers
- `ahk/`: AutoHotkey and MouseGestureL config for Windows
- `powershell/`: PowerShell helpers used from WSL during setup
- `rust/`: source for the `dotfiles` CLI
- `command/dotfiles`: built CLI artifact committed to the repo; do not edit this file directly
- `init.sh`: top-level setup entrypoint that runs the CLI

## How This Repo Works
- The Rust CLI performs setup tasks such as symlink creation, installer execution, and shell setup.
- Files under `home/` are the source of truth. They are linked into the user's real home directory by `dotfiles init`.
- `init.sh` and `rust/src/commands/init.rs` have system-level side effects:
  - package installation
  - symlink creation in `$HOME`
  - default shell changes
  - Windows-side WezTerm link creation when running under WSL
- Neovim plugins are configured as separate Lua modules under `home/config/nvim/lua/plugins/`.

## Common Tasks
- Adjust shell aliases, PATH, completions, or interactive shell behavior in `home/zshrc`
- Add or modify Neovim plugin specs under `home/config/nvim/lua/plugins/`
- Update Neovim keymaps, options, or autocmds under `home/config/nvim/lua/core/`
- Tweak WezTerm behavior under `home/config/wezterm/`
- Update package bootstrap scripts under `sh/installers/`
- Modify Rust CLI behavior in `rust/src/` and keep tests passing
- Update Windows automation in `ahk/` or `powershell/`

## Editing Rules
- Prefer the smallest change that satisfies the request.
- Do not edit `command/dotfiles` by hand. If the Rust CLI changes and the user wants the compiled artifact updated, rebuild it from `rust/`.
- Do not casually modify `home/config/nvim/lazy-lock.json`. Only update it when intentionally changing plugin versions or after a plugin management action that requires the lockfile to change.
- Treat `init.sh`, `rust/src/commands/init.rs`, installer scripts, and shell startup files as high-impact files.
- Avoid introducing new external dependencies unless the user asked for them or they are clearly required.
- Preserve existing naming and layout conventions.
- When changing shell or Lua config, match the local style.

## High-Risk Files And Areas
- `init.sh`
- `rust/src/commands/init.rs`
- `home/zshrc`
- `sh/installers/**/*`
- `powershell/mklink.ps1`
- `home/config/nvim/lazy-lock.json`
- `command/dotfiles`

Prefer extra caution, narrower edits, and explicit verification notes when touching these.
